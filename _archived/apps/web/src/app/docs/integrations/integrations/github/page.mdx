## prerequisites

- a github repo to receive issues
- a personal access token (classic or fine-grained) with `issues:write` on the target repo
- somewhere to host uploaded files — github does not accept direct binary uploads into issue bodies, so you need a storage target (s3, r2, a gist, a release asset, your own cdn)

## install

```bash
npm install remediate @octokit/rest
```

## environment variables

```bash
# .env
GITHUB_TOKEN=ghp_xxxxxxxxxxxx
GITHUB_OWNER=your-org
GITHUB_REPO=your-repo
```

## client — mount the widget

```tsx
<Remediate endpoint="/api/feedback" />
```

## server — api route

format the feedback with `toGithubIssue`, upload each file to your storage of choice, rewrite the placeholder tokens in the body with the final urls, then create the issue via octokit.

```ts
// app/api/feedback/route.ts
import { parseFeedback } from "remediate/server";
import { toGithubIssue } from "remediate/github";
import { Octokit } from "@octokit/rest";

const octokit = new Octokit({ auth: process.env.GITHUB_TOKEN });

export async function POST(req: Request) {
  const feedback = await parseFeedback(req);
  const issue = toGithubIssue(feedback);

  // upload files somewhere github can reach (e.g. a gist, release asset,
  // s3 bucket, or your own cdn) then swap the placeholders in issue.body
  let body = issue.body;
  for (const file of issue.files) {
    const url = await uploadSomewhere(file.content, file.filename);
    body = body.replace(file.placeholder, url);
  }

  const created = await octokit.issues.create({
    owner: process.env.GITHUB_OWNER!,
    repo: process.env.GITHUB_REPO!,
    title: issue.title,
    body,
    labels: issue.labels,
  });

  return Response.json({
    ok: true,
    id: feedback.submission.id,
    githubIssue: created.data.number,
  });
}
```

## output shape

- `title` — derived from the first note, annotation, or item caption
- `body` — markdown with `![screenshot](…)` and `[download](…)` tags, plus an environment footer. placeholders look like `{{screenshot-<id>}}`
- `labels` — always includes `feedback`, plus any annotation priority tags (e.g. `urgent`, `high`)
- `files` — blob + placeholder pairs you resolve before creating the issue
